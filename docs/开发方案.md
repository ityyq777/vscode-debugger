这是一个关于构建 **VSCode 自动化调试代理系统** 的完整开发方案。本方案旨在解决闭源 Agent 无法直接操作 VSCode API 以及无法接收异步回调的难题。

---

### 一、 系统架构概览

系统采用 **C/S (客户端/服务端)** 架构，利用 **阻塞式 HTTP 请求** 实现双向同步。

#### 1. 组件角色
*   **vscode-debugger-proxy (服务端)**:
    *   部署形态：VSCode Extension。
    *   职责：随 VSCode 启动，监听本地端口；将 HTTP 请求转化为 VSCode API 调用；监听底层调试事件并阻塞请求响应。
*   **vscode-debug-agent-skills (知识库/协议)**:
    *   部署形态：Prompt 提示词 / MCP Server 定义文件。
    *   职责：教会闭源 Agent 如何调用 Proxy API，定义调试工作流。
*   **Claude Code for VSCode (客户端)**:
    *   职责：读取 Skills，根据用户意图发起 HTTP 请求，解析返回的调试状态。

#### 2. 核心交互模式：同步阻塞
Agent 发送“继续/单步”指令时，Proxy **不立即返回**，而是挂起 HTTP 连接，等待 VSCode 触发 `stopped` 事件（如命中断点）后，再将当前状态返回给 Agent。这解决了 Agent 无法建立 WebSocket 连接的问题。

### 二、 模块一：`vscode-debugger-proxy` 开发详解

#### 1. 技术栈
*   语言：TypeScript
*   依赖：`express` (HTTP服务), `vscode` (Extension API)
*   端口：建议 `4711` (可配置)

#### 2. 核心机制实现

**A. 状态管理器**
用于协调 HTTP 请求与 VSCode 异步事件。

```typescript
// StateManager.ts
import * as vscode from 'vscode';

type DebugStatus = 'running' | 'stopped' | 'terminated';

interface DebugState {
    status: DebugStatus;
    reason?: string; // 'breakpoint', 'step', 'exception'
    line?: number;
    file?: string;
}

// 全局状态
let currentState: DebugState = { status: 'stopped' };
let resolver: ((value: DebugState) => void) | null = null;

// 阻塞等待函数：由 HTTP 接口调用
export async function waitForState(expectedStatus: DebugStatus, timeout = 30000): Promise<DebugState> {
    if (currentState.status === expectedStatus) return currentState;
    
    return new Promise((resolve) => {
        resolver = (newState) => {
            if (newState.status === expectedStatus || newState.status === 'terminated') {
                resolve(newState);
                resolver = null;
            }
        };
        // 超时保护
        setTimeout(() => resolve({ ...currentState, status: 'timeout' }), timeout);
    });
}

// 状态更新函数：由事件监听器调用
export function updateState(newState: Partial<DebugState>) {
    currentState = { ...currentState, ...newState };
    if (resolver) resolver(currentState);
}
```

**B. 底层事件监听**
使用 `DebugAdapterTracker` 拦截 DAP 协议消息，这是获取“断点命中”、“单步结束”等精确事件的唯一可靠方式。

```typescript
// extension.ts
import * as vscode from 'vscode';
import { updateState } from './StateManager';

export function activate(context: vscode.ExtensionContext) {
    // 注册 DAP 消息拦截器
    context.subscriptions.push(
        vscode.debug.registerDebugAdapterTrackerFactory('*', {
            createDebugAdapterTracker(session: vscode.DebugSession) {
                return {
                    onDidSendMessage: (message: any) => {
                        if (message.event === 'stopped') {
                            // 调试器停止（断点/单步/异常）
                            updateState({
                                status: 'stopped',
                                reason: message.body.reason,
                                // 此处可解析 stackTrace 获取 file/line
                            });
                        } else if (message.event === 'continued') {
                            updateState({ status: 'running' });
                        } else if (message.event === 'terminated') {
                            updateState({ status: 'terminated' });
                        }
                    }
                };
            }
        })
    );

    startHttpServer(); // 启动 HTTP 服务
}
```

**C. HTTP API 定义**

| Endpoint | Method | 功能 | 阻塞特性 | 实现逻辑 |
| :--- | :--- | :--- | :--- | :--- |
| `/launch` | POST | 启动调试 | 是 | 调用 `startDebugging`，等待 `stopped` (entry/breakpoint) |
| `/breakpoints` | POST | 设断点 | 否 | 调用 `addBreakpoints` |
| `/control` | POST | 流程控制 | **是** | 调用 `next/continue`，等待 `stopped` |
| `/variables` | GET | 获取变量 | 否 | 调用 `customRequest('variables')` 或读缓存 |
| `/status` | GET | 查询状态 | 否 | 立即返回当前 `currentState` |

**D. 关键接口代码示例**

```typescript
// server.ts (简化版)
app.post('/control', async (req, res) => {
    const { action } = req.body; // 'next', 'continue', 'stepIn'
    
    // 1. 发送控制指令
    const commandMap = {
        next: 'workbench.action.debug.stepOver',
        continue: 'workbench.action.debug.continue',
        stepIn: 'workbench.action.debug.stepInto'
    };
    
    await vscode.commands.executeCommand(commandMap[action]);
    
    // 2. 阻塞等待调试器再次停止 (关键点)
    const finalState = await waitForState('stopped');
    
    // 3. 返回结果给 Agent
    res.json({
        success: true,
        status: finalState.status,
        reason: finalState.reason,
        message: `Execution paused at ${finalState.file}:${finalState.line}`
    });
});
```

### 三、 模块二：`vscode-debug-agent-skills` 设计详解

由于 Agent 是闭源的，此模块通常以 **Prompt 文件** 或 **MCP Server 配置** 的形式存在。核心是明确告诉 Agent：**“这是一个阻塞式调试工具”**。

#### 1. Skill 定义示例

```markdown
## Tool: vscode_debugger_proxy

### Base URL
`http://localhost:4711`

### Description
A proxy to control the VSCode debugger. 
**CRITICAL**: Most operations are **synchronous and blocking**. 
When you send a 'next' or 'continue' command, the request will **wait** until the debugger pauses again (e.g., hits a breakpoint or finishes a step). Do not assume it failed if it takes time to respond.

### Actions & Endpoints

1. **Set Breakpoint**
   - `POST /breakpoints`
   - Body: `{ "file": "absolute_path", "lines": [10, 25] }`
   - Returns: Success/Failure.

2. **Start Debugging**
   - `POST /launch`
   - Body: `{ "program": "${workspaceFolder}/main.js" }`
   - **Blocking**: Waits until the program starts and hits "stopOnEntry" or the first breakpoint.

3. **Step Over (Next)**
   - `POST /control` Body: `{ "action": "next" }`
   - **Blocking**: Waits until the line execution finishes and debugger pauses at the next line.

4. **Continue**
   - `POST /control` Body: `{ "action": "continue" }`
   - **Blocking**: Waits until the next breakpoint is hit or program terminates.

5. **Inspect Variables**
   - `GET /variables`
   - Returns: JSON of current scope variables.

### Workflow Example
1. User asks to debug a logic error on line 50.
2. Agent calls `POST /breakpoints` for line 50.
3. Agent calls `POST /launch`.
   -> Request hangs...
   -> Program hits breakpoint.
   -> Request returns `{ "status": "stopped", "line": 50 }`.
4. Agent calls `GET /variables` to check data.
5. Agent calls `POST /control { "action": "next" }` to step through.
   -> Request hangs...
   -> Debugger steps to line 51.
   -> Request returns `{ "status": "stopped" }`.
6. Agent analyzes the change and suggests a fix.
```

### 四、 完整交互流程演示

1.  **场景**: 用户要求 Agent 调试死循环。
2.  **Agent (决策)**: 根据 Skills，先设置断点。
    *   Agent -> Proxy: `POST /breakpoints { file: "loop.js", lines: [10] }`
    *   Proxy -> Agent: `{ "success": true }`
3.  **Agent (决策)**: 启动调试。
    *   Agent -> Proxy: `POST /launch { program: "loop.js" }`
    *   *系统状态: VSCode 启动调试，运行至第 10 行暂停。*
    *   Proxy (DAP 监听器): 捕获 `stopped` 事件。
    *   Proxy -> Agent: `{ "status": "stopped", "reason": "breakpoint", "line": 10 }` (请求解除阻塞)
4.  **Agent (决策)**: 查看变量，判断循环变量 `i` 的值。
    *   Agent -> Proxy: `GET /variables`
    *   Proxy -> Agent: `{ "i": 100, "max": 10 }`
5.  **Agent (决策)**: 发现 `i` 一直递增未重置，尝试单步执行确认逻辑。
    *   Agent -> Proxy: `POST /control { action: "next" }`
    *   *系统状态: VSCode 执行一行代码。*
    *   Proxy (DAP 监听器): 捕获 `stopped` (step) 事件。
    *   Proxy -> Agent: `{ "status": "stopped", "line": 11 }`
6.  **Agent (结论)**: 确认逻辑错误，生成修复代码。

### 五、 开发路线图

#### Phase 1: 最小可行性产品 (MVP)
*   **目标**: 实现“启动 -> 断点 -> 单步”的基础闭环。
*   **开发**:
    1.  搭建 VSCode 插件框架，启动 Express Server。
    2.  实现 `DebugAdapterTracker` 监听 `stopped` 事件。
    3.  实现阻塞式的 `waitForState` 逻辑。
    4.  实现 `/control` 和 `/breakpoints` 接口。
    5.  编写基础 Skills Prompt，手动测试 API 是否符合预期。

#### Phase 2: 数据增强与容错
*   **目标**: 能够读取变量，处理超时。
*   **开发**:
    1.  实现 `/variables` 接口（利用 `activeDebugSession.customRequest` 请求 scopes/variables）。
    2.  增加超时机制：如果调试运行超过 30 秒无断点，返回 timeout 状态。
    3.  完善 Skills 文档，加入错误处理指引。

#### Phase 3: 高级特性
*   **目标**: 多线程支持、监视表达式、调用栈回溯。
*   **开发**:
    1.  实现完整的调用栈获取。
    2.  支持 Evaluate 表达式（`POST /evaluate`）。
    3.  将 Proxy 封装为标准 MCP Server（如果 Claude Code 支持标准 MCP 协议，这将极大提升体验）。

通过此方案，你成功地在闭源 Agent 和 VSCode 之间建立了一个同步、可控、透明的调试通道。
