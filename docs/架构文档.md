# VSCode Debugger Proxy 系统架构文档

## 一、系统概览

### 1.1 背景与目标

本系统解决闭源 Agent（如 Claude Code for VSCode）无法直接操作 VSCode API 的难题。通过 HTTP 同步阻塞机制，Agent 可以远程控制 VSCode 调试器进行代码调试。

### 2.4 DapTracker - DAP 事件拦截器

#### 什么是 DapTracker

DapTracker（Debug Adapter Tracker）是 VSCode 扩展中用于拦截调试适配器协议（Debug Adapter Protocol, DAP）消息的组件。

#### 为什么需要 DapTracker

VSCode 调试 API 本身不提供直接的回调机制来感知"断点命中"等事件。通过 DapTracker 拦截 DAP 协议消息，我们可以获取：
- 调试器何时停止（stopped 事件）
- 停止的原因（breakpoint、step、exception 等）
- 当前的堆栈帧信息（file、line、frameId）

#### 核心职责

```mermaid
flowchart LR
    A[用户操作] --> B[VSCode]
    B --> C[DAP Protocol]
    C --> D[DapTracker]
    D --> E[StateManager]
    E --> F[Agent 感知]
```

1. **监听用户操作**：检测用户对调试器的操作（继续、单步、停止等）
2. **检测状态变化**：检测调试器状态变化（断点命中、异常、单步完成等）
3. **同步状态更新**：将这些事件同步到 StateManager

#### DAP 事件类型

| 事件 | 触发时机 | 说明 |
|------|----------|------|
| `stopped` | 断点命中、单步完成、异常、暂停 | 调试器停止运行 |
| `continued` | 用户点击继续或单步后 | 调试器恢复运行 |
| `terminated` | 会话结束 | 调试会话被终止 |
| `exited` | 进程退出 | 被调试进程退出 |
| `thread` | 线程变化 | 线程创建或退出 |

#### 停止原因（stopped.reason）

| 原因 | 说明 |
|------|------|
| `breakpoint` | 命中断点 |
| `step` | 单步操作完成 |
| `exception` | 抛出异常 |
| `entry` | 停在程序入口 |
| `pause` | 用户暂停 |

#### 代码位置

`vscode-debugger-proxy/src/DapTracker.ts`

#### 关键实现

```typescript
// 注册 DapTracker 工厂，拦截所有调试类型的 DAP 消息
vscode.debug.registerDebugAdapterTrackerFactory('*', {
    createDebugAdapterTracker(session) {
        return {
            onWillStartSession: () => { /* 调试开始 */ },
            onWillStopSession: () => { /* 调试结束 */ },
            onDidSendMessage: (message) => {
                if (message.event === 'stopped') {
                    // 拦截停止事件，获取断点位置
                    const frame = await session.customRequest('stackTrace', {...});
                    updateState({ file: frame.source.path, line: frame.line });
                }
            }
        };
    }
});
```

## 三、API 接口

### 3.1 接口列表

| 接口 | 方法 | 阻塞 | 功能 |
|------|------|------|------|
| `/health` | GET | 否 | 健康检查 |
| `/status` | GET | 否 | 获取调试状态 |
| `/breakpoints` | POST | 否 | 设置断点 |
| `/launch` | POST | **是** | 启动调试 |
| `/relaunch` | POST | **是** | 重新启动调试 |
| `/wait` | POST | **是** | 等待断点命中 |
| `/control` | POST | **是** | 单步/继续 |
| `/variables` | GET | 否 | 获取变量 |
| `/stacktrace` | GET | 否 | 获取栈帧 |
| `/stop` | POST | 否 | 停止调试 |

### 3.2 请求响应格式

**断点设置请求**:
```json
{
  "/path/to/app.py": [29, 30],
  "/path/to/utils.py": [10]
}
```

**Wait 响应**:
```json
{
  "success": true,
  "status": "stopped",
  "reason": "breakpoint",
  "file": "/path/to/app.py",
  "line": 29,
  "frameId": 4,
  "message": "Stopped at /path/to/app.py:29 (breakpoint)"
}
```

## 四、调试时序流程

### 4.1 完整调试流程

```mermaid
sequenceDiagram
    participant Agent
    participant Proxy as vscode-debugger-proxy
    participant VSCode
    participant Debugger

    Note over Agent: 读取端口文件<br/>.vscode/debug-proxy.port

    Agent->>Proxy: POST /breakpoints<br/>{"app.py": [29]}
    Proxy->>VSCode: addBreakpoints()
    VSCode-->>Proxy: 断点已设置
    Proxy-->>Agent: {success: true}

    Agent->>Proxy: POST /launch<br/>{"config": "Python: Flask"}
    Proxy->>VSCode: startDebugging()
    VSCode->>Debugger: 启动调试进程

    Note over Debugger: 程序执行中...

    Debugger->>Proxy: 命中断点 29
    Proxy->>Agent: POST /wait 返回<br/>{status: "stopped", frameId: 4}

    Agent->>Proxy: GET /variables?frameId=4
    Proxy->>VSCode: scopes + variables
    VSCode-->>Proxy: {user_id: "1", name: "Alice"}
    Proxy-->>Agent: {variables: {...}}

    Agent->>Proxy: POST /control<br/>{"action": "next"}
    Proxy->>VSCode: stepOver
    VSCode->>Debugger: 单步执行

    Debugger->>Proxy: 停在下一行
    Proxy->>Agent: {status: "stopped", line: 30}

    Agent->>Proxy: POST /control<br/>{"action": "continue", "wait": true, "timeout": -1}

    Note over Debugger: 继续执行...

    Debugger->>Proxy: 命中下一个断点
    Proxy->>Agent: {status: "stopped", line: 50}
```

### 4.2 单终端 Agent 死锁风险

```mermaid
sequenceDiagram
    participant Agent
    participant Proxy
    participant Flask

    Note over Agent: 单终端 Agent (会死锁)

    Agent->>Proxy: POST /wait
    Note over Proxy: 阻塞中...

    Agent->>Flask: curl http://localhost:5000/users/1
    Note over Flask: 无法处理请求<br/>因为 Agent 终端被阻塞

    Flask-->>Agent: (等待响应)
    Note over Agent: 死锁！Agent 无法处理 Flask 的响应

    Proxy-->>Agent: (永远等待)
```

**解决方案**: 单终端 Agent 必须使用 `/wait` + 外部触发机制

```mermaid
sequenceDiagram
    participant Agent
    participant Proxy
    participant User
    participant Flask

    Agent->>Proxy: POST /wait
    Note over Proxy: 阻塞等待...

    User->>Flask: 手动访问 URL
    Flask->>Flask: 处理请求
    Flask->>Debugger: 触发断点
    Debugger->>Proxy: 断点命中
    Proxy-->>Agent: {status: "stopped"}

    Note over Agent: 成功！终端未被阻塞
```

### 4.3 状态同步与 frameId 校验

```mermaid
sequenceDiagram
    participant Agent
    participant Proxy
    participant User
    participant DapTracker

    Agent->>Proxy: POST /wait
    Proxy-->>Agent: {frameId: 4, line: 29}

    Note over User: 用户手动 continue

    User->>VSCode: 点击继续按钮
    VSCode->>DapTracker: continued 事件
    DapTracker->>Proxy: 更新状态 frameId=null

    Agent->>Proxy: GET /variables?frameId=4
    Proxy->>Proxy: 校验 frameId
    Note over Proxy: frameId 不匹配！4 ≠ null

    Proxy-->>Agent: {success: false, error: "Frame has changed"}
```

## 五、Agent 使用流程

### 5.1 标准工作流

```mermaid
flowchart TD
    A[读取 .vscode<br/>debug-proxy.port] --> B[连接 Proxy<br/>http://localhost:xxxx]
    B --> C[设置断点<br/>/breakpoints]
    C --> D[启动调试<br/>/launch]
    D --> E{程序停止？}
    E -->|是| F[获取变量<br/>/variables]
    F --> G[分析代码]
    G --> H{需要继续？}
    H -->|单步| I[/control next]
    I --> E
    H -->|到下一断点| J[/control continue]
    J --> E
    E -->|超时| K[处理超时]
    H -->|完成| L[结束]
```

### 5.2 典型调试场景

**场景：调试 Flask 应用**

```bash
# 1. 读取端口
cat .vscode/debug-proxy.port  # 输出: 54321

# 2. 设置断点
curl -X POST http://localhost:54321/breakpoints \
  -H "Content-Type: application/json" \
  -d '{"d:/project/app.py": [29]}'

# 3. 启动调试
curl -X POST http://localhost:54321/launch \
  -H "Content-Type: application/json" \
  -d '{"config": "Python: Flask"}'

# 4. 等待断点命中（阻塞）
curl -X POST http://localhost:54321/wait

# 5. 查看变量
curl "http://localhost:54321/variables?frameId=4"

# 6. 单步执行
curl -X POST http://localhost:54321/control \
  -H "Content-Type: application/json" \
  -d '{"action": "next"}'

# 7. 继续到下一断点
curl -X POST http://localhost:54321/control \
  -H "Content-Type: application/json" \
  -d '{"action": "continue", "wait": true, "timeout": -1}'
```

## 六、安全与限制

### 6.1 安全机制

1. **frameId 校验**: 所有操作携带 frameId，防止状态不一致
2. **调试器状态检查**: 操作前验证调试器是否活动
3. **文件存在性检查**: 设置断点前验证文件路径

### 6.2 限制

| 限制 | 说明 |
|------|------|
| 端口文件位置 | 仅支持第一个 workspace folder |
| 断点格式 | 仅支持源码行断点，不支持条件断点 |
| 多调试会话 | 仅支持单个活动调试会话 |
| 跨窗口 | 不支持控制其他 VSCode 窗口的调试器 |
