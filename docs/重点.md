断点是随时可以增加和取消的，跟所调试的程序运行到哪里了没有关系
对于agents来说它可以一下子打了很多断点，但这个时候还没有请求过来，这是没关系的agent也不需要等，只需要agent再请求wait接口，与proxy建立通信连接，等proxy察觉断点命中了再返回
proxy还应该同步调试程序启动之前已经存在的断点/和用户在程序运行时自己想打的断点

我认为/wait应该返回当前的frameId，然后请求变量的时候也应该带上这个请求参数
agents在请求control/variables都应该携带上某个凭证，让proxy去做校验，确保没有被用户干预到，如果与当前凭证预期不一样就应该及时返回异常信息给agent，这个凭证用frameId比较合适

请求所有的接口（除了/lunch）之前应该确保调试器是启动的状态，对于/lunch接口如果调试器是关闭的，那就做启动工作，如果是已经开启的那就直接返回success:True（关闭调试器同理）
